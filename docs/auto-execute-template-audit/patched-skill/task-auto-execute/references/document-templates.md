# Software Delivery Standard Document Templates

Use these templates when generating a project-local delivery standard pack. Replace every placeholder with project-specific facts. If a fact is unavailable from PRD, UI, code, or user-provided sources, write a blocker row with status `BLOCKED_BY_MISSING_SOURCE`; do not invent details.

This skill is documentation-only. These templates describe standards and future execution requirements. They must not cause the current skill run to execute code, start apps, click pages, call APIs, mutate a database, create `results/*.json`, write `latest/*HANDOFF.md`, or mark any task `PASS`.

## 2. `<slug>-development-standard.md`

```markdown
# <Project Name> Development Standard

## Non-Negotiable Rules
- Tie every code change to a requirement ID and task ID.
- Preserve user-provided PRD and UI references as the highest project evidence after direct instructions.
- Keep changes small, reviewable, and reversible.
- Do not touch production data, real payment, or real cloud writes unless explicitly authorized.
- Do not claim implementation complete without matching test evidence.

## Lifecycle Standard
| Phase | Required Work Product | Exit Criteria |
| --- | --- | --- |
| Intake | source inventory, blocker list | all PRD/UI/code sources mapped |
| Design | requirement/UI/API/DB matrices | P0/P1 scope traceable |
| Implementation | task-local code changes | allowed files only, no hidden scope |
| Verification | tests and evidence | relevant standard-test-plan rows pass |
| Handoff | result JSON + HANDOFF | next worker can resume from evidence |
| Final gate | final acceptance report | fail closed on missing P0 proof |

## Product Quality Standard
| Quality Area | Development Rule | Required Evidence |
| --- | --- | --- |
| Functional suitability | P0/P1 behavior matches PRD | requirement matrix + task result |
| Performance | P0 flows avoid obvious latency/regression | performance note or test where applicable |
| Compatibility | supported browsers/devices/runtime documented | page/runtime smoke evidence |
| Usability | page states, labels, errors, loading, empty states complete | UI screenshots and owner scenarios |
| Reliability | retries, errors, persistence, idempotency where needed | error-path tests |
| Security | auth, access control, validation, secrets protected | security checklist/results |
| Maintainability | local patterns, clear modules, no needless abstraction | code review notes |
| Portability | environment config and local setup documented | start/run commands |
| Safety/Risk | external side effects guarded | local-only adapter proof |

## Frontend Standard
- Every P0 page has route, UI reference, states, controls, API/data dependencies, and screenshot target.
- Every page defines loading, empty, error, success, unauthorized, and persisted-data states when applicable.
- Every P0 click has an expected route, modal, toast, state transition, API call, or validation message.
- Text must fit containers at supported viewport sizes.
- Accessibility basics: labels, focus order, keyboard access, contrast, and status messages where applicable.

## Backend/API Standard
- Every API has method/path, auth rule, request schema, response schema, validation rules, error envelope, and persistence side effects.
- Every P0 mutation has DB write proof and independent readback proof.
- External services must use local/mock/sandbox adapters by default.
- Error responses must be user-safe and developer-debuggable without leaking secrets.

## Frontend/Backend Contract Standard
- Every frontend API caller must map to a backend contract row.
- Request and response shapes must be validated by tests or typed fixtures.
- Contract drift blocks PASS.

## Database Standard
- Every table/collection used by P0 flows must have owner, seed data, mutation rules, readback rules, and cleanup policy.
- Test data must be identifiable and safe to inspect.

## External Data Standard
- Every spreadsheet, Feishu Bitable, SaaS storage, or external data target must have a row-level field mapping matrix.
- Every outbound payload must be validated before write and captured as evidence during later execution.
- Every table/entity that uses `unique_key` must prove create, update/upsert, duplicate-run idempotency, and independent readback.
- Failed collection or failed external writes must not pollute metrics with `0`; failed numeric values must remain null/None plus explicit failure status.
- Failed external writes must be cached locally and replayed by a later task before final PASS.
- Missing real credentials must route to local mock/sandbox proof unless the user explicitly authorizes real external writes.

## Forbidden Actions
- No real payment.
- No production data.
- No real cloud write unless explicitly authorized.
- No destructive cleanup of user work.
- No broad refactor unrelated to the task.
- No pure PASS without evidence.
```
